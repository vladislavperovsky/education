# Практическое упражнение: Выбери архитектуру для ShopWave

> **Тема:** 2.1 — Архитектурные паттерны
> **Инструменты:** draw.io (app.diagrams.net) или Miro, шаблон `ecommerce_bounded_contexts.drawio`

---

## Контекст

Вы — системный аналитик в e-commerce компании **ShopWave**. Три года назад стартап из 3 разработчиков запустил интернет-магазин на монолите (один процесс, одна БД). Сейчас:

- **50 000 заказов/день** (рост с 1 000 три года назад)
- **40 разработчиков** в 5 командах
- **Релиз — раз в 2 недели**, потому что все ждут общего релизного окна
- **Баг в модуле скидок положил весь сайт** в Чёрную пятницу — потеряли 2 млн выручки за час
- **Нельзя масштабировать только каталог** — в Чёрную пятницу нагрузка на каталог выросла в 10 раз, пришлось поднимать весь монолит (дорого)
- **Технологический стек заморожен** — переход на новую версию фреймворка тянет изменения во всех модулях

Бизнес требует: «Сделайте так, чтобы команды могли выкатывать изменения независимо, а сайт не падал из-за одного модуля».

---

## Задание

### Шаг 1. Анализ проблем монолита (5 мин)

Опишите **минимум 3 ключевые проблемы** текущей архитектуры ShopWave, используя терминологию из лекции:

- Связанность (coupling)
- Узкое место деплоя
- Ограничение масштабирования
- Единая точка отказа
- Сложность кодовой базы

**Формат:** список, каждый пункт — 1-2 предложения.

| Критерий | Монолит (оставить) | Модульный монолит | Микросервисы | Event-driven | Гибрид |
|----------|-------------------|-------------------|-------------|-------------|--------|

| Независимость команд | | | | | |
| Масштабирование под нагрузкой | | | | | |
| Отказоустойчивость | | | | | |
| Стоимость инфраструктуры и DevOps | | | | | |

**Как заполнять:** для каждого критерия поставьте `+` (преимущество), `-` (недостаток) или `0` (нейтрально).

**Итоговый выбор:** «Я выбираю _______________, потому что: (2-3 ключевых аргумента)»

### Шаг 3. Декомпозиция на bounded contexts (20 мин)

Выделите **4-6 bounded contexts** для ShopWave. Для каждого контекста укажите:

1. Название контекста
2. Ключевая ответственность (1 предложение)
3. Какие данные этот контекст хранит (2-3 основных сущности)
4. Какая команда им владеет (условно: Команда А, Команда Б…)

**Исходные домены ShopWave (из которых нужно выделить контексты):**

| Домен | Что делает |
|-------|-----------|
| **Каталог товаров** | Управление карточками товаров, категориями, ценами, остатками на витрине |
| **Корзина и заказы** | Сбор корзины, оформление заказа, статусы заказа, история |
| **Платёжи** | Приём платежей, возвраты, интеграция с платёжными шлюзами |
| **Склад** | Управление складскими остатками, сборка заказов, отгрузка |
| **Доставка** | Расчёт стоимости, выбор службы, трекинг посылок |
| **Клиенты** | Профили пользователей, адреса, история покупок, сегментация |
| **Маркетинг** | Промокоды, скидки, акции, персональные рекомендации |
| **Уведомления** | Email, SMS, push — транзакционные и маркетинговые |
| **Аналитика** | Дашборды, отчёты по продажам, воронка конверсии |
| **Поддержка** | Тикеты, обращения клиентов, возвраты, база знаний |

**Нарисуйте диаграмму bounded contexts:**

- Каждый контекст — прямоугольник с названием
- Стрелки между контекстами — потоки данных
- Подпишите стрелки: что передаётся и как (REST / События / Batch)

**Подсказки по определению границ:**
- Если две части системы меняют одни и те же данные — они в одном контексте
- Если слово значит разное в разных отделах — это разные контексты (напр. «Заказ» для Продаж ≠ «Заказ» для Склада)
- Одна команда должна владеть одним bounded context

### Шаг 4. Сравнение с эталоном (10 мин)

Модератор покажет эталонное решение. Сравните и ответьте на вопросы:

- Какие контексты вы объединили, а эталон разделил (и наоборот)?
- Почему вы провели границу иначе? Какие были аргументы?
- Какие связи между контекстами вы пропустили или добавили лишних?

## Критерии оценки

| # | Критерий | Вес |
|---|----------|-----|
| 1 | Определены 3+ ключевых проблемы текущей архитектуры | обязательно |
| 2 | Таблица trade-offs заполнена (все 6 критериев) | обязательно |
| 3 | Выбор паттерна обоснован (2+ аргумента) | обязательно |
| 4 | Выделены 4-6 bounded contexts с обоснованием границ | обязательно |
| 5 | Диаграмма показывает связи между контекстами с подписями (что + как) | обязательно |
| 6 | Есть легенда (типы связей: синхронные / асинхронные / batch) | рекомендуется |
