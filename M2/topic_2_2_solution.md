## Описание участников интеграции

Опираясь на архитектуру bounded contexts из задания 2.1, описываю участников интеграции для сценария оформления заказа:

| Система | Роль в процессе | Что передаёт | Что получает |
|---------|-----------------|--------------|--------------|
| **Client (Витрина)** | Точка входа пользователя, инициатор процесса | Данные корзины, адрес доставки, способ оплаты | Подтверждение заказа, номер заказа, статус |
| **API Gateway** | Единая точка входа для всех запросов, маршрутизация | Нормализованные запросы к backend-сервисам | Агрегированные ответы от микросервисов |
| **Order & Payments** | Основной оркестратор процесса, управление заказом и платежами | События: OrderCreated, PaymentConfirmed, OrderCompleted | Статусы от Fulfilment, Catalog (резервирование товара) |
| **Catalog** | Проверка доступности товара, резервирование | Подтверждение наличия товара, актуальные цены | Запрос на резервирование товаров из заказа |
| **Fulfilment (Склад)** | Резервирование товара, сборка заказа, управление отгрузкой | События: ItemsReserved, OrderPicked, ReadyForShipment | OrderCreated (через events), задача на сборку |
| **Fulfilment (Доставка)** | Организация доставки, трекинг посылки | Статус доставки: InTransit, Delivered | Данные для отгрузки: адрес, вес, габариты |
| **Customer Engagement** | Уведомления клиента на всех этапах | Email/SMS/Push уведомления | События обо всех изменениях статуса заказа |
| **Analytics** | Сбор данных для отчётности и прогнозирования | — | Все события из процесса заказа (batch) |

### Ключевые наблюдения:
- **Order & Payments** — system of record для состояния заказа
- **API Gateway** — синхронный слой для клиентских запросов
- **Event-driven** коммуникация между внутренними сервисами для асинхронной обработки

## Сравнение трёх подходов

Анализирую trade-offs для контекста ShopWave с учётом специфики:
- Система склада иногда отвечает медленно (нестабильное время ответа)
- Пиковые нагрузки (Чёрная пятница, распродажи)
- Ошибки нельзя терять
- Клиенту важен мгновенный ответ о принятии заказа

| Подход | Как работает в ShopWave | Плюсы | Минусы | Когда подходит |
|--------|-------------------------|-------|--------|----------------|
| **REST / RPC (синхронный)** | API Gateway → Order → Catalog → Fulfilment. Каждый вызов ждёт ответа предыдущего | Простота реализации<br>Понятная логика<br>Мгновенная валидация | Клиент ждёт медленный Fulfilment<br>Каскадные сбои (падение Fulfilment → падение Order)<br>Не масштабируется при росте нагрузки | Только для критически важных синхронных операций (проверка наличия товара перед добавлением в корзину) |
| **Сообщения / События (async)** | Order публикует OrderCreated → Fulfilment подписан → обрабатывает асинхронно. Брокер (RabbitMQ/Kafka) гарантирует доставку | Клиент получает ответ мгновенно<br>Изоляция отказов (Fulfilment упал → Order работает)<br>Естественное масштабирование<br>Retry и гарантия доставки | Eventual consistency (статус обновляется позже)<br>Сложность отладки и трейсинга<br>Требует брокера сообщений | **Основной сценарий** для заказа → склад → доставка |
| **Batch / ETL (пакетная обработка)** | Order записывает в БД → раз в N минут джоб читает новые заказы → отправляет в Fulfilment пачкой | Эффективность для больших объёмов<br>Простая интеграция со старыми системами<br>Снижает нагрузку на API | Задержка обработки (минуты/часы)<br>Неприемлемо для UX (клиент ждёт)<br>Сложность обработки ошибок | Только для Analytics и неприоритетных задач (отчёты, дата-синхронизация) |

### Вывод по сравнению:
Для критичного пути "заказ → склад → доставка" нужен **гибридный подход**:
- **REST** — для валидации корзины и создания заказа (Order ↔ Catalog)
- **Events** — для асинхронной обработки склада и доставки
- **Batch** — только для Analytics

## Выбор целевой стратегии

### Решение: Event-Driven Architecture с синхронными элементами

**Обоснование выбора:**

1. **Мгновенный ответ клиенту**: Order & Payments синхронно принимает заказ, проверяет товары через REST-вызов к Catalog, создаёт транзакцию платежа. Клиент сразу получает подтверждение с номером заказа.

2. **Асинхронная обработка склада**: После создания заказа Order публикует событие `OrderCreated` в брокер (Kafka/RabbitMQ). Fulfilment подписан на это событие и обрабатывает его независимо. Если склад медленный или временно недоступен — брокер гарантирует доставку события позже.

3. **Изоляция отказов**: Падение Fulfilment не влияет на Order & Payments. Заказ сохранён, клиент получил подтверждение, обработка на складе произойдёт когда система восстановится.

4. **Масштабирование**: В пиковые нагрузки Order быстро публикует события, а Fulfilment обрабатывает очередь с комфортной скоростью. Можно добавить больше consumer'ов Fulfilment для параллельной обработки.

5. **Уведомления**: Customer Engagement подписан на все события (`OrderCreated`, `ItemsReserved`, `ReadyForShipment`, `Delivered`) и отправляет клиенту SMS/email на каждом этапе.

## Ответы на вопросы

### 1. Где нужен мгновенный ответ, а где можно обработать позже?

**Мгновенный ответ (синхронно):**
- **Проверка наличия товара** (Catalog) — клиент не должен добавить в корзину то, чего нет
- **Создание заказа** (Order & Payments) — клиент должен сразу увидеть номер заказа
- **Инициация платежа** — пользователь ждёт редирект на платёжную систему

**Можно обработать позже (асинхронно):**
- **Резервирование на складе** — клиент уже получил подтверждение, склад резервирует в течение 5-10 минут
- **Сборка заказа** — происходит в течение нескольких часов
- **Создание задачи на доставку** — формируется после готовности заказа
- **Уведомления** — отправляются фоном, не влияют на основной флоу
- **Аналитика** — данные собираются batch-процессом раз в час

### 2. Какая система является system of record?

**Order & Payments** — единственный system of record для:
- Состояния заказа (статус: created → paid → reserved → picked → shipped → delivered)
- Финансовых транзакций
- Связи заказа с клиентом

**Почему Order, а не Fulfilment?**
- Заказ существует независимо от склада (может быть цифровым товаром)
- Один заказ может включать товары с разных складов
- Клиент взаимодействует с заказом, а не со складом

**Fulfilment** управляет только своей частью — физическим исполнением (резерв, сборка, отгрузка).

### 3. Что будет при временной недоступности склада?

**Сценарий:**
1. Клиент оформляет заказ → Order сохраняет заказ в БД со статусом `CREATED`
2. Order публикует событие `OrderCreated` в Kafka/RabbitMQ
3. **Fulfilment недоступен** (техническая авария, деплой, перегрузка)
4. Брокер сообщений **сохраняет событие в очереди** и будет пытаться доставить его позже

**Результат:**
- Клиент получил подтверждение заказа и номер
- Заказ не потерян, событие в очереди
- Когда Fulfilment восстановится → он прочитает очередь и обработает все накопленные заказы
- Customer Engagement отправит уведомление клиенту о статусе

**Защитные меры:**
- **Dead Letter Queue (DLQ)** — если событие не обработалось после N попыток → идёт в отдельную очередь для разбора
- **Monitoring & Alerts** — если очередь растёт → алерт команде операторов
- **Graceful degradation** — на витрине показываем "Заказ принят, обработка может занять до 30 минут"

### 4. Как не потерять заказ при сбое?

**Защита на уровне сохранения заказа:**

Когда клиент оформляет заказ, Order & Payments сохраняет его в базу данных внутри транзакции. Если сервис падает в процессе сохранения — транзакция откатывается, и клиент получит ошибку. Это значит, что заказ либо полностью сохранён, либо не сохранён вообще — промежуточного состояния нет.

**Защита на уровне передачи событий:**

После успешного сохранения заказа Order публикует событие `OrderCreated` в брокер сообщений (Kafka или RabbitMQ). Брокер сохраняет это событие на диск и реплицирует на несколько серверов. Даже если Order & Payments упадёт сразу после публикации — событие останется в брокере и будет доставлено Fulfilment'у.

**Защита от потери события при обработке:**

Fulfilment читает событие из очереди, обрабатывает его (резервирует товар на складе), и только после успешной обработки отправляет брокеру подтверждение (acknowledgement). Если Fulfilment упал до подтверждения — брокер считает событие необработанным и отдаст его другому экземпляру Fulfilment или повторит попытку позже.

**Защита от дублирования:**

Каждое событие содержит уникальный `orderId`. Перед обработкой Fulfilment проверяет: "Я уже обрабатывал заказ с таким ID?". Если да — игнорирует событие. Это называется идемпотентность и защищает от ситуации, когда одно и то же событие обрабатывается дважды из-за retry.

**Итого:**
Заказ защищён на всех этапах: транзакция в БД → надёжное хранение в брокере → retry при сбоях → защита от дублей.

## Риски и меры защиты

| Риск | Вероятность | Влияние | Мера защиты |
|------|-------------|---------|-------------|
| **Потеря событий при сбое брокера** | Низкая | Критичное | Kafka с репликацией (3 реплики), Outbox Pattern в Order |
| **Дублирование обработки** | Средняя | Среднее | Идемпотентность consumer'ов (проверка event_id) |
| **Недоступность Fulfilment >15 мин** | Средняя | Высокое | Retry с backoff, DLQ, автоотмена заказа через 30 мин |
| **Переполнение очереди в пик** | Высокая | Среднее | Autoscaling consumers, приоритезация, мониторинг queue depth |
| **Рассинхронизация статусов** | Средняя | Среднее | Order как system of record, eventual consistency принята как trade-off |
| **Каскадный сбой при падении Order** | Низкая | Критичное | Circuit Breaker на API Gateway, health checks, graceful degradation |

## Заключение

Выбранная **Event-Driven Architecture с синхронными элементами** оптимально решает задачи ShopWave:

- **Мгновенный ответ клиенту** — через синхронный REST для Order  
- **Изоляция отказов** — падение Fulfilment не ломает Order  
- **Масштабирование** — асинхронная обработка справляется с пиковыми нагрузками  
- **Надёжность** — события не теряются благодаря брокеру и правильной обработке ошибок  
- **Прозрачность** — клиент видит актуальные статусы через уведомления  

Архитектура защищена от основных проблем (медленный склад, пиковые нагрузки, сбои) через комбинацию паттернов: retry, timeout, idempotency, DLQ и monitoring.
